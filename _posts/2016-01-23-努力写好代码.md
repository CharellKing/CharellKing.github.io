最近，我一直在优化自己的代码，倒不是因为自己的代码性能太差，而是因为代码风格太差。我以前一直以为写代码，就像写白话一样，表达要清楚；即使一个不懂编程的人，看了代码，就像看报，立刻能明白代码的意思。现在我觉得错了，其实写代码和行文一样，切记啰嗦，要一刀见血；大文豪海明威写的小说，别人评价他的小说是简洁，有战地记者的风范，如果抖动整本书，不会有一个多余的字掉落出来，写代码也一样。写代码应该像写唐宋之后的古文一样明了；我不推崇以前战国时的那种写法，以为那时候的古文太过复杂了，一句古文可以解释成多种意思，存在歧义，想想论语就知道了。然而我们读几遍韩愈、苏东坡的散文之后，就能整体把握它们的道理，这是恰到好处，既保证了简洁，又不失易懂。


## 行数最少

保证行数最少，不是刻意的用分隔符把本该写多行的代码垒到一行，就是要把握分寸；写过的代码，你要多回过头来想想，类似的代码之后你也会不断的写，如果它是一个“瘤子”，那这个瘤子就永远存在你的代码里，一直得不到根除。我最近写python比较多，说说我的代码‘瘤子’有哪些：

+ 对不同的key进行统计

``` python
if "key" not in a:
    a["key"] = 0
a["key"] += 1
```

由于python没有三目运算，却有if...else...; 有时候追求简洁，会丢失一些不值一提的性能问题，那是值得的； 但是大多时候简洁意味少bug、易读、甚至性能提升。上面四行代码解决的问题，我现在用一行了，上面的代码似乎易懂些,但是下面的理解起来也不难。

``` python
a["key"] = (a["key"] + 1) if "key" in a else 0
```

+ 对list里面的每个元素进行计算

``` python
other_list = []
for elem in one_list:
    other_list.append(elem + 2)
one_list = other_list
```

``` python
one_list = [elem + 2 for elem in one_list]
```

+ 将list里面的元素映射到dict的key中

``` python
one_dict = {}
for elem in one_list:
    one_dict[elem] = 0
```

``` python
one_dict = {elem: 0 for elem in one_list}
```

+ list添加多个元素

``` python
one_list.append("a")
one_list.append("b")
one_list.append("c")
one_list.append("d")
```

``` python
one_list.extend(["a", "b", "c", "d"])
```

代码的行数多了，看的特累，因为人的习惯是按行来阅读；而真正意识到这个问题，我是从王垠的博客中学来；把代码行数尽可能写的最少，是件很吃力的事情。

## 命名简洁
过去我写代码追求变量名称一目了然，尽量让变量函数命名如其意。比如有一张学生信息表，我要从以学号排序来获取学生信息、以成绩排序来获取学生信息；它们的变量名称，我可能就这样了命名了：students_sort_by_no、students_sort_by_grade。这样的变量名太长了，由于我限定的代码列数不超过120，一个有关的复杂表达式就超过了指定的列，于是得换行；而换行是特别破坏代码美观的，所以特别纠结。现在想出办法就是把变量字符数减少，比如：stus_sort_no、stus_sort_grade，把student给换成缩写，当然这样的缩写很不伦不类，但是如果你的代码全篇都是这样的缩写，就会给读者带来定式，stu就代表students；你就不能一会儿stu，一会儿stud了。

变量名太复杂，影响阅读；太过简单（如a、b、c），影响其意思。我建议变量名称的单词个数不能超过3个，而函数名称的单词个数不能超过5个；还有要尽量使用约定俗成的单词缩写，而这些约定俗成的缩写往往来自于标准库，而标准库是每个码农都会用的，都很熟悉；比如src对应source、dst对应destination、str对应string等等，有必要多学习标准库的命名习惯。


## 调试信息
我好多时候，认为大量的打印信息，会影响程序的整洁；所以一般调试完毕之后，会删除，现在觉得是有问题的。我好多次运行程序，程序运行了很久，没有一点提示信息，还以为程序死循环了，特不踏实。好多程序将系统日志分为7个等级，我觉得太复杂了，但是起码得4个：

+ [info] 就是我上面提到的那种情况，记录程序运行情况、进度，可以让你吃一颗定心丸， 用sys.stdout.write或者print

+ [debug] 调试的时候用，打印变量的信息，用sys.stdout.write或者print

+ [warn] 有问题，但是有时候可以忽略的，建议以错误的方式输出，用sys.stderr.write。

+ [error] 有严重问题了，一定要解决。以错误的方式输出，同样用sys.stderr.write。

这招是从大牛的代码里面学来，大牛的工具做的很棒，虽然代码写的一般般，还是有了一些可取的地方。

``` python
sys.stdout.write("[info]xxxx")
sys.stdout.write("[debug]xxxx")
sys.stderr.write("[warn]xxxx")
sys.stderr.write("[error]xxxx")
```

## 代码规范

在中国，小公司制作的代码开发规范，不够详细、漏洞百出；大公司制作的代码规范，一直深藏在内部；目前我就只知道流出的华为C++代码规范；其实也有很多大公司和小公司，就直接使用google的代码规范了，google的内部开发语言有c++、java、python、js、go。在网上可以看到c++、java、python、js的代码规范；而且c++、python、js有对应的规范检测工具，分别对应cpplint、pylint、jslint，配置到自己IDE里面，只要你写一行不符合规范的代码，马上就会给你提示。我在想如果BAT三家公司公开自己的编码规范，并且有配套的代码检测工具，就会引来很多公司、机构使用，每年有大量的码农跳槽到这些公司，那么好多码农就不需要提前熟悉公司里面的代码规范，就可以直接上岗了；其实好多人写代码的风格某些部分会一直保留下去，即使换了环境；比如一个按照百度的代码规范写了5年的码农，跑到了阿里，我相信阿里的代码库里面充斥着他的部分百度风格代码；我在想这是不是同化了？一种编程规范很优秀很强大，会在不同的地方生根发芽。


## 多划分模块函数

记得有人批评过我，说我的代码太难读，要不断的跳转，眼睛都跳花了；我自认为自己不是一个有函数封装癖的人，在上一家公司，有个员工特喜欢研究设计模式，恨不得把helloworld这条打印语句进行封装了；最后这个员工被辞退，他写的代码，没有一个人敢维护。一个人被辞退，说明他不是一个牛B的人，被别人看不起；反之，只有变得很牛B，才能让天下的人瞧得起你。我们一直觉得写代码和读代码是一个相反的过程，写的过程中，我们强调模块化、封装，意味会有很多类、函数，这样造成我们在读代码的时候，要借助工具的跳转功能；而读代码了，我们希望像读文章一样，代码的执行顺序和代码的行文顺序一样，恨不得所有的代码从文本的第一行开始，直到文本的最后一行结束，这样眼也不花了，腰也不痛了。

大型的系统有几十万行、上百万行代码，要是这样一行行读下去，等到读完的那一刻，人都呕吐好几回了。同时也说明这个人没有掌握阅读代码之法；阅读代码，我觉得先要把所有的文件大致过一遍，把握整体的框架、大致功能模块，不要追究里面的的细节，然后在针对感兴趣的部分，详细研究。在阅读函数的过程，某些函数出现很多次了，你看了一遍之后，之后就不用看了；如果你没有封装代码，你要把大量相同的代码给用视野过滤掉。从这个角度讲，函数封装是有利于代码阅读的。

我一般遇到如下的情况会划分模块函数：

+ 代码确实可以复用

为了减少代码的行数、模块化；这种情况是理所当然的。

+ 一个函数超过了一个屏幕版面

一个函数超过一个屏幕版面，意味着需要翻页，看起来会很费劲；如果要联系上下文，意味着你的鼠标要来回不停的滚动; 这时候需要拆分部分代码到另一个函数，即使这个函数不能被复用，也是应该的。

+ 脱离业务的代码

由于经常要写相同的代码，有些代码和业务无关，所以我自己弄了一个自己的python库，这个库目前在一点点的扩大。比如我要经常读写CSV文件、经常需要过滤字符串、经常要把大量的数据分进程批次处理、经常要自动发邮件，经常要执行mysql等等。

+ 逻辑太复杂了

逻辑太复杂了，把大量的代码揉在一起，很容易出bug；我往往是这样，把逻辑进行拆分，只专心一小部分，写一个小函数，调通这个小函数，这个小部分就完成了；然后把它忘到九霄云外，就这样慢慢的吞噬复杂的问题.


## 即时审视自己的代码

我最近优化自己的代码，花了很大的代价；由于代码量很多、整个系统这半年来不断的新增功能；忘记了很多代码的功能，需要花几分钟推敲；整个系统，改动之后，带来一些bug，需要大量的调试。

其实我在几个月之前，就知道系统的扩展方向了，以及功能的增长点在哪儿；比如之前由于需求的变动，旧有的系统要继续运行， 我在数据库中保留了这个字段，新建了另一个类似的字段。而旧有的字段一直是我的眼中刺，早就想拔除，那时候的确很忙，我借口是将来有空弄吧；系统不断增加功能，现在做起来有点积重难返了；而且对代码有点不熟悉了，整个优化起来就味同嚼蜡。其实如果当时发现问题苗头，趁热打铁进行根除，是最好的；虽然会花点时间，比起调试代码花的时间，根本不值得一提，况且精简的系统，bug量会更少。
